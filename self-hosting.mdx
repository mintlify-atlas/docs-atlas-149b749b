---
title: "Self-hosting"
description: "Deploy OwlMetry on your own Ubuntu VPS in 15–20 minutes. Full control, no data leaving your servers."
---

This guide walks you through deploying OwlMetry on a self-hosted Ubuntu VPS. By the end you'll have:

- API server at `api.yourdomain.com`
- Ingest endpoint at `ingest.yourdomain.com`
- Web dashboard at `yourdomain.com`

<Warning>
  **Alpha software** — OwlMetry is in early alpha. APIs, schemas, and configuration may change without notice. It is not yet production-ready.
</Warning>

## Requirements

- Ubuntu 24.04 LTS VPS (1 GB RAM minimum, 25 GB disk recommended)
- A domain pointed to the VPS with 3 A records: root (`@`), `api`, and `ingest`
- SSH access to the server as root

<Note>
  The setup script creates a 2 GB swap file. This is required for Next.js builds on 1 GB VPS instances — the build will run out of memory without it.
</Note>

## Setup

<Steps>
  <Step title="Run the setup script">
    Run this on your VPS as root. It installs Node.js 22, PostgreSQL 16, nginx, pm2, pnpm, and creates the `owlmetry` database.

    ```bash
    curl -fsSL https://raw.githubusercontent.com/Jasonvdb/owlmetry/main/deploy/setup-ubuntu-vps.sh -o setup-ubuntu-vps.sh
    bash setup-ubuntu-vps.sh
    ```

    <Warning>
      **Save the `DATABASE_URL` printed at the end of the script.** The database password is generated once and will not be shown again.
    </Warning>

    The script is idempotent — safe to run again if interrupted.
  </Step>

  <Step title="Clone and build">
    ```bash
    cd /opt/owlmetry
    git clone https://github.com/Jasonvdb/owlmetry.git .
    pnpm install
    pnpm build
    ```
  </Step>

  <Step title="Configure environment">
    Copy the example file and edit it with your values:

    ```bash
    cp .env.example .env
    ```

    ```bash .env
    DATABASE_URL=postgresql://owlmetry:<password>@localhost:5432/owlmetry
    JWT_SECRET=<random-64-char-string>
    PORT=4000
    HOST=0.0.0.0
    CORS_ORIGINS=https://yourdomain.com
    MAX_DATABASE_SIZE_GB=8
    RESEND_API_KEY=<your-resend-key>
    EMAIL_FROM=noreply@yourdomain.com
    COOKIE_DOMAIN=.yourdomain.com
    ```

    Generate a random JWT secret:

    ```bash
    openssl rand -base64 48
    ```

    | Variable | Required | Description |
    |---|---|---|
    | `DATABASE_URL` | Yes | PostgreSQL connection string — use the value printed by the setup script |
    | `JWT_SECRET` | Yes | Random string for signing JWTs. Generate with `openssl rand -base64 48` |
    | `PORT` | No | API server port (default: `4000`) |
    | `HOST` | No | Bind address (default: `0.0.0.0`) |
    | `CORS_ORIGINS` | Yes | Your dashboard URL, e.g. `https://yourdomain.com` |
    | `MAX_DATABASE_SIZE_GB` | No | Auto-prune events when the database exceeds this size in GB. Default is `0` (disabled) |
    | `RESEND_API_KEY` | No | [Resend](https://resend.com) API key for sending verification emails. If unset, verification codes print to the server console — fine for single-user setups |
    | `EMAIL_FROM` | No | From address for verification emails (default: `noreply@owlmetry.com`) |
    | `COOKIE_DOMAIN` | Yes\* | Cookie domain for cross-subdomain auth, e.g. `.yourdomain.com`. Required when the API and dashboard are on different subdomains |
  </Step>

  <Step title="Run migrations">
    ```bash
    pnpm db:migrate
    ```

    This creates all tables and event partitions. Your first user account is created automatically when you log in via the dashboard or CLI.

    <Warning>
      Do **not** run `pnpm dev:seed` in production. The seed script creates a demo user with deterministic API keys that are committed to the git repository and publicly known.
    </Warning>
  </Step>

  <Step title="Configure nginx">
    Create the nginx configuration with three server blocks:

    ```bash
    cat > /etc/nginx/sites-available/owlmetry << 'NGINX'
    # OwlMetry API (agent keys, dashboard API, auth)
    server {
        listen 80;
        server_name api.yourdomain.com;

        location / {
            proxy_pass http://127.0.0.1:4000;
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection 'upgrade';
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            proxy_cache_bypass $http_upgrade;
        }
    }

    # OwlMetry Ingest (SDK client keys)
    server {
        listen 80;
        server_name ingest.yourdomain.com;

        client_max_body_size 2m;

        location / {
            proxy_pass http://127.0.0.1:4000;
            proxy_http_version 1.1;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }
    }

    # OwlMetry Web Dashboard
    server {
        listen 80;
        server_name yourdomain.com;

        location / {
            proxy_pass http://127.0.0.1:3000;
            proxy_http_version 1.1;
            proxy_set_header Upgrade $http_upgrade;
            proxy_set_header Connection 'upgrade';
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            proxy_cache_bypass $http_upgrade;
        }
    }
    NGINX
    ```

    Enable the configuration and reload nginx:

    ```bash
    ln -sf /etc/nginx/sites-available/owlmetry /etc/nginx/sites-enabled/owlmetry
    rm -f /etc/nginx/sites-enabled/default
    nginx -t
    systemctl reload nginx
    ```
  </Step>

  <Step title="Start services with pm2">
    Create the pm2 ecosystem file:

    ```bash
    cat > /opt/owlmetry/ecosystem.config.cjs << 'PM2'
    module.exports = {
      apps: [
        {
          name: "owlmetry-api",
          script: "apps/server/dist/index.js",
          cwd: "/opt/owlmetry",
          env: {
            NODE_ENV: "production",
          },
        },
        {
          name: "owlmetry-web",
          script: "node_modules/.bin/next",
          args: "start",
          cwd: "/opt/owlmetry/apps/web",
          env: {
            NODE_ENV: "production",
            PORT: 3000,
          },
        },
      ],
    };
    PM2
    ```

    Start both services and persist them across reboots:

    ```bash
    cd /opt/owlmetry
    pm2 start ecosystem.config.cjs
    pm2 save
    ```

    Verify both services are running:

    ```bash
    pm2 status
    ```

    You should see `owlmetry-api` and `owlmetry-web` both with status `online`.
  </Step>

  <Step title="Set up SSL">
    Choose one option based on whether you're using Cloudflare:

    <Tabs>
      <Tab title="Cloudflare (recommended)">
        If you're using Cloudflare in proxy mode (orange cloud), Cloudflare handles SSL termination. nginx only needs to listen on port 80 — no certbot required.

        Configure Cloudflare:

        1. Add your domain to Cloudflare.
        2. Create 3 A records pointing to your VPS IP, all with **Proxy enabled** (orange cloud):
           - `@` (root) → VPS IP
           - `api` → VPS IP
           - `ingest` → VPS IP
        3. Set SSL/TLS mode to **Full** (not Full Strict, since you're not using certbot).
        4. Lock down the VPS firewall to only accept HTTP/HTTPS from [Cloudflare IPs](https://www.cloudflare.com/ips/).
      </Tab>
      <Tab title="certbot (no Cloudflare)">
        If you're not using Cloudflare, install certbot and obtain certificates for all three subdomains:

        ```bash
        apt install -y certbot python3-certbot-nginx
        certbot --nginx -d yourdomain.com -d api.yourdomain.com -d ingest.yourdomain.com
        ```
      </Tab>
    </Tabs>
  </Step>

  <Step title="Configure the firewall">
    Restrict incoming connections to SSH via Tailscale and HTTP/HTTPS via Cloudflare IPs only:

    ```bash
    ufw default deny incoming
    ufw default allow outgoing
    ufw allow in on tailscale0 to any port 22 proto tcp

    # Allow Cloudflare IPs only
    for cidr in $(curl -s https://www.cloudflare.com/ips-v4/); do
      ufw allow from "$cidr" to any port 80,443 proto tcp
    done
    for cidr in $(curl -s https://www.cloudflare.com/ips-v6/); do
      ufw allow from "$cidr" to any port 80,443 proto tcp
    done

    ufw --force enable
    ```

    <Note>
      If you're not using Cloudflare and not using Tailscale, replace the SSH rule with `ufw allow 22/tcp` to allow SSH from any IP.
    </Note>
  </Step>

  <Step title="Verify the deployment">
    Test the API directly on the server:

    ```bash
    curl -s http://localhost:4000/health
    # Expected: {"status":"ok"}
    ```

    Test nginx routing from a machine that can reach the server:

    ```bash
    curl -s https://api.yourdomain.com/health
    curl -s https://ingest.yourdomain.com/health
    ```

    Open `https://yourdomain.com` in a browser to see the dashboard.
  </Step>
</Steps>

## Updating

To pull the latest code and rebuild:

```bash
cd /opt/owlmetry
git pull
pnpm install
pnpm build
pnpm db:migrate
pm2 restart all
```

## Database maintenance

- **Partitions** are auto-created on server startup for the current month plus 2 months ahead.
- **Auto-pruning** runs hourly when `MAX_DATABASE_SIZE_GB` is set — drops oldest monthly partitions first.
- **Backups**: `pg_dump owlmetry > backup.sql`
- **Size check**: `psql -c "SELECT pg_size_pretty(pg_database_size('owlmetry'));"`

## Troubleshooting

```bash
# Check service logs
pm2 logs owlmetry-api --lines 50
pm2 logs owlmetry-web --lines 50

# Check nginx configuration
nginx -t

# Check database connectivity
psql $DATABASE_URL -c "SELECT 1;"

# Check disk space
df -h

# Restart services
pm2 restart all
```
