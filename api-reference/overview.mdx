## Base URL

OwlMetry is self-hosted, so your base URL is the domain where you deployed the API server:

```
https://api.yourdomain.com
```

During local development the server runs on port `4000` by default:

```
http://localhost:4000
```

All versioned endpoints are prefixed with `/v1`. The health check endpoint sits at the root.

## Health check

Use the health endpoint to verify the server is running. It requires no authentication.

```bash
curl https://api.yourdomain.com/health
```

```json
{ "status": "ok" }
```

## Request format

All request and response bodies are JSON. Set the `Content-Type` header on any request with a body:

```
Content-Type: application/json
```

## Pagination

Endpoints that return lists of records — `/v1/events`, `/v1/apps/:id/users`, `/v1/projects/:projectId/metrics/:slug/events`, and `/v1/teams/:teamId/audit-logs` — use cursor-based pagination.

Pass these query parameters to control the result set:

| Parameter | Type | Description |
|-----------|------|-------------|
| `limit` | integer | Maximum number of records to return per page |
| `cursor` | string | Opaque cursor returned by the previous page; omit on the first request |

The response includes a `next_cursor` field when more records exist. Pass it as `cursor` on your next request to fetch the following page. When `next_cursor` is absent or `null`, you have reached the last page.

```bash
# First page
curl -H "Authorization: Bearer owl_agent_..." \
  "https://api.yourdomain.com/v1/events?limit=50"

# Next page
curl -H "Authorization: Bearer owl_agent_..." \
  "https://api.yourdomain.com/v1/events?limit=50&cursor=<next_cursor>"
```

## Rate limits

The API enforces a token-bucket rate limit per authenticated identity:

| Property | Value |
|----------|-------|
| Bucket capacity | 100 requests |
| Refill rate | 10 requests per second |
| Scope | Per API key, per user session, or per IP (unauthenticated) |

When you exceed the limit, the server responds with `429 Too Many Requests` and a `Retry-After: 1` header. Back off for at least one second before retrying.

Verification code requests have an additional limit: a maximum of **5 codes per email address per hour**.

## Error responses

All error responses use a consistent JSON shape:

```json
{
  "error": "A human-readable description of what went wrong"
}
```

Common HTTP status codes:

| Status | Meaning |
|--------|---------|
| `400` | Bad request — missing or invalid parameters |
| `401` | Unauthorized — missing, invalid, or expired credentials |
| `403` | Forbidden — authenticated but lacking the required role or permission |
| `404` | Not found — resource does not exist or you do not have access to it |
| `429` | Too many requests — rate limit exceeded |
| `500` | Internal server error |

<Note>
  404 is also returned when you lack access to a resource, to avoid leaking information about resource existence.
</Note>
